# Emacboros - Agentic Emacs Workspace

* Overview
This repository contains an agentic Emacs workspace designed for privacy-preserving, extensible development and AI assistance. It follows the philosophy of local-first computing, containment first security, and human agency empowerment.

**Core Architecture:**
- Local-first Emacs configuration with agent orchestration
- Containerized deployment with hardened security
- Distributed hardware design for resilience
- No backdoors or telemetry - fully auditable

* Security Features
Emacboros implements multiple layers of security:

**Container Hardening:**
- Preflight mount audit prevents container escapes
- Capability dropping (only NET_RAW, NET_BIND_SERVICE retained)
- Writable path restrictions on dangerous system locations
- Read-only root filesystem overlay

**Operational Security:**
- Air-gapped systems design
- Identity management with aliases
- Audit logging of all file operations and command executions
- File guard protecting critical paths (see below)

**File Guard Protected Categories:**
The file guard (~file_guard.el~) enforces write, replace, and append
restrictions on critical system files. Protected paths are organized
into two tiers:

***Always Protected (cannot be bypassed):**
1. *Agent prompt files* (~agents.d/<name>/prompt.org~) — prevents agents from rewriting their own or other agents' prompts
2. *Shared context* (~agents.d/base_context.org~) — prevents context tampering
3. *HISTORY.log files* — append-only; overwrite and replace are blocked

***Conditionally Protected (relaxed in self-modification mode):**
4. *Emacs Lisp source* (~init.el~, ~init.d/*.el~) — prevents tool tampering
5. *Container config* (~Containerfile~, ~emacboros.sh~) — prevents container escape
6. *Git hooks* (~.git/hooks/*~) — prevents scheduled execution

**Self-Modification Mode:**
When ~my-gptel--guard-allow-self-modification~ is set to non-nil,
categories 4–6 are relaxed, allowing trusted agents to modify Emacs
Lisp source, container configuration, and git hooks during development.

Categories 1–3 remain protected regardless of this setting. An agent
should never silently rewrite its own prompt or the shared context.

To enable for the current session:
#+BEGIN_SRC elisp
  (setq my-gptel--guard-allow-self-modification t)
#+END_SRC

To enable persistently via Emacs customize:
- ~M-x customize-variable RET my-gptel--guard-allow-self-modification RET~
- Toggle to ~t~, then save for future sessions

*Warning:* This mode is intended for development sessions only. Do
NOT enable for CTF or untrusted-content sessions. This is a software
check (defense-in-depth); container mounts should also be read-only
for categories 4–6 in production.

* Agent Architecture
The system supports dynamic agent personas through gptel integration:

- **Agent Discovery**: Automatically detects agents in `agents.d/<name>/prompt.org`
- **Dynamic Loading**: Seamlessly expands Org-mode includes for each agent profile
- **State Management**: Tracks current agent and associated file paths
- **Memory Tools**: Agent-specific memory summarization available via C-c m

**Key Emacs Components:**
- `agent_loader.el` - Dynamic agent discovery and loading
- `delegate_tool.el` - Multi-agent delegation capabilities
- `gptel_setup.el` - GPTEL backend configuration
- Various tools for code execution, file operations, and system tasks

* Development Workflow
**Prerequisites:**
- Podman or Docker (for container builds)
- Emacs 28+ with native compilation support
- Python 3.8+

**Building Containers:**
#+BEGIN_SRC bash
./containers/build.sh
#+END_SRC

This script builds all images defined in the containers/images/ directory.

**Running Emacs:**
The main container entry point runs:
1. Preflight mount audit (security check)
2. Permission fix verification
3. Emacs startup with loaded configuration

**Local Development:**
- Clone this repository
- Run `emacs --load /path/to/init.el` to start the workspace
- Use `C-c a` to load different agent personas
- Access tools via gptel interface

* Philosophy and Principles

**1. Privacy First**
- All processing occurs locally
- No cloud dependencies or telemetry
- Local-first design ensures data sovereignty

**2. Containment First**
- Every autonomous capability has a kill switch
- Audit logs for all operations
- Defined scope with clear boundaries

**3. Precision Over Speed**
- Measured approaches to system changes
- Cost of mistakes considered in terms of lives
- Conservative security posture

**4. Human Agency**
- Systems empower individuals, not control them
- No backdoors or hidden capabilities
- Fully auditable and inspectable code
