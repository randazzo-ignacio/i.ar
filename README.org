# i.ar

A self-modifying AI operating environment built in Emacs, running in a hardened Podman container, powered by local LLMs via Ollama. No cloud. No telemetry. No backdoors.

* Quickstart

** Prerequisites
- Podman (container runtime)
- Emacs 28+ (for local development without container)
- Ollama running on a reachable host (local or WireGuard mesh)
- A personalization repository (see below)

** Clone

#+begin_src bash
# Just want to use the tool (no personalization submodule):
git clone https://github.com/randazzo-ignacio/i.ar.git

# Want to work on i.ar or understand the codebase (includes personalization submodule):
git clone --recursive https://github.com/randazzo-ignacio/i.ar.git
#+end_src

The =personalization/= submodule contains knowledge bases, per-agent tasks, and audit logs. Users should create their own personalization repo with their own knowledge and agent configurations. See =personalization/= for the default setup.

** Build

#+begin_src bash
cd i.ar
./containers/build.sh
#+end_src

** Run

There is one entry point: =iar.sh=. It has two modes -- interactive (default) and loop (=--loop=).

Both modes require =--personalization= pointing to a directory with =knowledge/=, =tasks/=, and =audit/= subdirectories.

*** Interactive Sessions (default mode)

Drops you into an Emacs gptel chat buffer with an agent loaded. You're the conductor -- you type, the agent responds, you direct the work.

#+begin_src bash
# Interactive session with self-modification (for development with darwin)
./utils/iar.sh \
  --self-modification \
  --personalization ~/repos/iar-personalization \
  --gptel-fork ~/repos/gptel

# Interactive session (no self-modification, load agents via C-c a inside Emacs)
./utils/iar.sh \
  --personalization ~/repos/iar-personalization \
  --gptel-fork ~/repos/gptel
#+end_src

*** Autonomous Loops (=--loop=)

Runs an agent for N cycles with cooldown between each. Sends Telegram notifications if =AGENT_TELEGRAM_BOT_TOKEN= and =AGENT_TELEGRAM_CHAT_ID= are set. Container names are auto-suffixed with PID so multiple loops can run in parallel without collisions.

#+begin_src bash
# Darwin autonomous loop (code changes, self-modification enabled)
./utils/iar.sh --loop \
  --self-modification \
  --personalization ~/repos/iar-personalization \
  --agent darwin \
  --gptel-fork ~/repos/gptel \
  --max-cycles 50

# Gardener autonomous loop (prompt refinement, no self-modification)
./utils/iar.sh --loop \
  --personalization ~/repos/iar-personalization \
  --agent gardener \
  --gptel-fork ~/repos/gptel \
  --max-cycles 10

# Playground loop on GPU model (sophon, 120B, 128K context)
./utils/iar.sh --loop \
  --personalization ~/repos/iar-personalization \
  --agent playground \
  --model nemotron-3-super:120b \
  --ctx 131072 \
  --ollama-host 10.66.0.5:11434 \
  --max-cycles 999 \
  --cooldown 300

# Playground loop on CPU model (daftpunk, 30B, 128K context)
./utils/iar.sh --loop \
  --personalization ~/repos/iar-personalization \
  --agent playground \
  --model granite4.1:30b \
  --ctx 131072 \
  --ollama-host 10.66.0.3:11434 \
  --max-cycles 999 \
  --cooldown 300
#+end_src

*** Model and Host Selection

The =--model= and =--ctx= flags control which LLM runs and how much context window it gets. These are critical for local models -- the KV cache scales linearly with context size, so a 120B model at 1M context would need ~320GB of RAM just for the cache.

| Flag | Default | Description |
|------+---------+-------------|
| =--model NAME= | =glm-5.2:cloud= | Model name (must be in =metaconfig/gptel.el= model list) |
| =--ctx N= | =1048576= (1M) | Max context window in tokens. Use 131072 (128K) or 262144 (256K) for local models |
| =--ollama-host HOST= | =10.66.0.5:11434= | Ollama API host:port |

Cloud models (suffix =:cloud=) handle context server-side and can use the default 1M. Local models need explicit =--ctx= to avoid OOM.

*** Running Multiple Sessions in Parallel

Container names auto-include the shell PID, so multiple =iar.sh= invocations can run simultaneously without name collisions:

#+begin_src bash
# Terminal 1: interactive mirror session
./utils/iar.sh --personalization ~/repos/iar-personalization --gptel-fork ~/repos/gptel

# Terminal 2: darwin loop running in background
./utils/iar.sh --loop --self-modification --personalization ~/repos/iar-personalization --agent darwin --gptel-fork ~/repos/gptel --max-cycles 50

# Terminal 3: playground loop on daftpunk
./utils/iar.sh --loop --personalization ~/repos/iar-personalization --agent playground --model granite4.1:30b --ctx 131072 --ollama-host 10.66.0.3:11434 --max-cycles 999
#+end_src

Use =podman ps= to see all running agent containers. Each is named =<agent>-loop-<PID>= (loop mode) or =iar-interactive-<PID>= (interactive mode).

** Inside Emacs

| Key | Action |
|-----|--------|
| C-c a | Load an agent (mirror, darwin, auditor, ctfwizard) |
| C-c k | Load a knowledge base (iar/, user/, infra/, etc.) |
| C-c p | Show prompt size (chars + approximate tokens) |
| C-c m | Summarize conversation to memory |

* Documentation

i.ar uses AI-first documentation. The knowledge base IS the documentation -- there is no separate wiki or architecture document.

To explore the project documentation interactively:

1. Load an agent (e.g., =C-c a mirror=)
2. Load the i.ar knowledge base (=C-c k iar/=)
3. Ask the agent about the architecture, design decisions, or code structure

The knowledge base files live in =personalization/knowledge/iar/= and cover:
- =architecture.md= -- Repository layout, container, network, security model
- =modules.md= -- Every Emacs Lisp module and what it does
- =agents.md= -- All agents, their roles, and the delegation system
- =tools.md= -- Every tool available to agents
- =philosophy.md= -- Design principles and decisions
- =usage.md= -- Detailed usage guide and personalization setup
- =workflow.md= -- Maintenance workflow (how to update docs after changes)
- =tool_gating.md= -- Security flags and power hierarchy

* Personalization

The =--personalization= flag mounts three subdirectories into the container:

- =knowledge/= -- Injectable knowledge bases (loaded via C-c k)
- =tasks/= -- Per-agent personal files (LOGS.md, SUMMARY.md, TODO.md, etc.)
- =audit/= -- Per-agent HISTORY.log files and global audit log

Create your own personalization repo with your own knowledge bases and agent configurations. Use the default =personalization/= submodule as a reference.
