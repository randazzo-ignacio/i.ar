* WHAT IGNISP IS

ignisp is a minimal Lisp derivative designed for maximum portability and longevity. It is not a product, a library, or a learning exercise. It is a life project -- a computational environment designed to survive decades and be rebuildable on any platform that exists or will exist.

The name comes from Ignacio (Nacho's name) + Latin *ignis* (fire). Fire is the spark that bootstraps everything -- the reducer is the spark, the language is the flame. It is both deeply personal (his name) and structurally meaningful (the bootstrap spark).

The core philosophy:

- **The spec is permanent. The implementation is disposable.** In 50 years, Layer 1 may have been rewritten 20 times in 20 host languages. The spec is what stays constant. The value is in the spec and the library, not in the implementation.

- **Three layers, three concerns.** Hardware abstraction (disposable), computational abstraction (permanent), and the language (permanent). Each layer has a clear job and a clear boundary.

- **Absolute portability over performance.** 100x slowdown is acceptable. The Layer 1 reducer must be small enough to rewrite in a day in any host language.

- **Force performance honesty.** The language should not hide computational cost behind lazy evaluation or runtime optimizations. If something is O(n^2), the programmer should know.

- **KISS and DRY.** Every added complexity must justify itself. Solo project principles that have paid dividends.

- **No hard dependencies.** If a dependency is small enough to rewrite in a day, it's acceptable. Otherwise, eliminate it.

- **Fully immutable.** No mutation anywhere -- no setq, no boxes, no mutable cells. State is threaded through recursion. Macros provide ergonomic syntax that expands to immutable code.

- **The 50-year horizon.** This is not a weekend project. It is a language to grow with, modify at will, and rebuild on any future platform. When Nacho is 70, he wants to still be using this language.

* THE ORIGIN STORY

This project emerged from a conversation about learning Common Lisp, which evolved into building a custom Lisp derivative. The key realizations that shaped the project:

1. Nacho has learned many languages (Lisp, Ruby, Bash, Python, Verilog, VHDL, C, C++, LaTeX, Ansible, and others) and developed a strong preference for customizability, elegance, and portability. He loves Lisp but never fully learned it because even with macros, CL's predefined choices don't make it *his*.

2. The project started as "a CL compiler" but evolved through conversation into something more radical: a portable Lisp built on lambda calculus, where the hardware layer is so small it can be rewritten for any host in a day.

3. The original design was metacircular (a minimal C kernel with arrays, arithmetic, and I/O, everything else in Lisp on top). This was rejected because of the metacircular bootstrap paradox (reader needs cons cells, cons cells need the reader) and because there was no boundary to optimize against. The three-layer architecture solves both problems.

4. The key insight: if the computational core is lambda calculus -- the mathematical foundation of computation -- then the language can be built on top in a way that is permanently portable. Lambda calculus has been studied for 80+ years and will outlive every hardware architecture. The hardware layer (the reducer) is the only thing that changes between platforms.

5. The hardware vision: the reducer is small enough to implement in Verilog, enabling a lambda calculus machine on FPGA. Graph reduction on FPGA is a well-studied technique. In 10-20 years, potentially a custom ASIC. This is a real goal, not science fiction.

* ARCHITECTURE: THREE-LAYER DESIGN

         This is where the programmer lives.
         ↓ compiled to
Layer 2: COMPUTATIONAL ABSTRACTION (permanent, the "assembly")
         Lambda calculus + native integers + I/O.
         Not human-writable. A compiler target.

         ↓ executed on
Layer 1: HARDWARE ABSTRACTION (disposable, rewrite per platform)

**Why three layers?** The metacircular approach (two layers) collapses the computational core and the language into one thing. Three layers gives each concern its own home and its own spec. Layer 2 is the boundary -- it has its own spec, its own identity. "Optimize the cons implementation" is a concrete task about Layer 2. The bootstrap is clean: Layer 2 is simple enough to generate from a script. No chicken-and-egg.

**Why lambda calculus for Layer 2?** It is permanent (studied 80+ years, will outlive every hardware architecture). It is minimal (three constructs: variables, abstraction, application, plus native integers and I/O). It separates computation from hardware (a paradigm shift changes Layer 1 but not Layer 2). It maps to any computational substrate (CPUs, GPUs, FPGAs, distributed systems). It is the honest answer to "what is the minimal computational substrate?"

* WHAT EACH LAYER CONTAINS

**Layer 1 (Hardware Abstraction) -- the reducer:**
- Representation of lambda terms (variables, abstractions, applications)
- Native integer values and operations (+, -, *, /, <, >, =)
- Character I/O (read-char, write-char)
- Beta reduction engine (eager / applicative order)
- Memory allocation and GC
- NO mutation. NO arrays. Pure functional.

**Layer 2 (Computational Abstraction) -- the permanent core:**
- Lambda abstraction: λx.M
- Application: M N
- Native integer operations (from Layer 1)
- I/O primitives (from Layer 1)
- Church-encoded data: booleans, pairs, lists
- Z combinator for recursion (no =define= in Layer 2)
- Thunked conditionals (no =if= special form in Layer 2)
- The ignisp compiler, reader, eval, and printer
- NO mutation. NO arrays. NO setq. Pure functional.

**Layer 3 (Ignisp) -- the language:**
- S-expression syntax (what the programmer writes)
- Special forms: if, quote, lambda, let, define, defmacro, begin (NO setq -- immutable language)
- Macros and macroexpansion
- Cons cells, strings, symbols, closures (Church-encoded in Layer 2)
- Standard library: list ops, string ops, math, I/O
- Ergonomic macros: =->= (thread-first), =loop=/=for= (comprehensions), =with-state= (state threading)
- Object system (CLOS-like, built on closures)
- Fully immutable. No mutation anywhere.

* KEY DESIGN DECISIONS

1. **Eager evaluation (applicative order).** Arguments are evaluated before the function is applied. Simpler to implement than lazy evaluation. No space leaks. Forces performance honesty: if you write an infinite list, it actually tries to evaluate it.

2. **Z combinator for recursion.** In pure lambda calculus with eager evaluation, functions can't refer to themselves by name. The Z combinator solves this. Every recursive function in Layer 2 is wrapped in Z. This is ugly for humans but Layer 2 is not human-writable -- the compiler handles it automatically. The alternative (adding =define= to Layer 2) was rejected because it breaks purity and every special form is something Layer 1 must understand.

3. **Church encoding for all data.** Cons cells, lists, booleans -- all Church-encoded in Layer 2. No arrays in any layer. Simplicity and purity over performance. If performance becomes a real problem, arrays can be added as a Layer 1 primitive later. The spec starts pure.

4. **Layer 2 is not human-writable.** This is a feature, not a bug. Layer 2 is mechanically generated by the compiler. Humans write ignisp (Layer 3); the compiler produces lambda terms.

5. **Fully immutable language.** No setq, no boxes, no mutable references anywhere. State is threaded through recursion. Macros provide ergonomic syntax. If shared mutable state is needed in the future, boxes can be added then. YAGNI.

6. **GC is a Layer 1 concern, not Layer 2.** The reducer owns all term memory. The root set is explicit: the current term being reduced plus any referenced terms. Mark-sweep over the term graph is straightforward (~50-100 lines of C). In the old metacircular architecture, GC was Layer 2 (Lisp code operating on arrays). In the new architecture, Layer 2 is lambda calculus -- it has no concept of memory management. The reducer allocates terms during beta reduction and must also collect them.

* MEMORY MANAGEMENT

- **Rust-style ownership was rejected** because it requires static analysis, restricts dynamic features (eval, runtime macros, arbitrary aliasing), and the result wouldn't be Lisp anymore.

- **Reference counting was rejected** because cycles leak (need separate cycle collector), every assignment needs inc/dec, and it's more code than mark-sweep for less benefit.

- **Mark-sweep GC was chosen** because it handles closures, cycles, eval, and dynamic dispatch with no language restrictions. The root set is explicit: the current term + referenced terms. The reducer owns all term memory.

- **Phase 1 (current): bump allocator, no GC.** malloc everything, never free. Acceptable for bootstrap and short programs. Add GC when memory pressure becomes real.

* BOOTSTRAPPING

The bootstrap problem: Layer 2 is not human-writable. The ignisp compiler (which translates Layer 3 to Layer 2) is itself written in... what? If it's written in ignisp, it needs to be compiled to Layer 2. But to compile it, you need the compiler. Chicken-and-egg.

The bootstrap solution: use a Python script as the bootstrap compiler.

1. Python implements a minimal lambda calculus reducer (Layer 1 in Python -- throwaway).
2. Python generates Layer 2 code that implements the ignisp core (reader, eval, basic stdlib).
3. Test the generated Layer 2 code on the Python reducer.
4. Write Layer 1 in C (the real reducer, ~300-500 lines).
5. Run the same Layer 2 code on the C reducer.
6. ignisp is now running. From this point, all development happens in ignisp (Layer 3). The Python script is discarded.

Bootstrap phases:
- Phase 0: Spec (1-2 weeks) -- write Layer 1, 2, 3 specs
- Phase 1: Bootstrap in Python (2-4 weeks) -- reducer + Layer 2 code generation
- Phase 2: C Reducer (1-2 weeks) -- Layer 1 in C, verify same behavior
- Phase 3: ignisp Core in ignisp (2-4 weeks) -- rewrite reader/eval/printer in ignisp, self-hosting
- Phase 4: Standard Library (ongoing) -- cond, and, or, let*, labels, string ops, list utilities, file I/O, error handling
- Phase 5: GC (1-2 weeks) -- mark-sweep in the C reducer
- Phase 6: Transpiler (long-term) -- compile ignisp to C, Python, or other targets
- Phase 7: Hardware (long-term dream) -- implement reducer in Verilog, run on FPGA

* SPECIFICATION PHILOSOPHY

Nacho wants specs written in a mathematical/structural style, not tied to concrete encodings. Character assignments are implementation parameters with a reference assignment. The structure is permanent; the encoding is disposable.

Example: Instead of writing "space (U+0020) is whitespace", write "the whitespace class separates tokens" and separately document "in the reference assignment, whitespace = space, tab, newline, CR".

The spec has three parts:
1. **Layer 1 Spec** -- what the reducer provides. The portable contract. Frozen.
2. **Layer 2 Spec** -- lambda calculus + ints + IO, evaluation rules, encoding of basic data. Frozen once stable.
3. **Layer 3 Spec** -- ignisp: types, special forms, reader syntax, evaluation model. Frozen once stable.
4. **Standard Library Spec** -- everything built on top. Grows forever.
