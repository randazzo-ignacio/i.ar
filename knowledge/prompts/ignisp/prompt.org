* WHAT IGNISP IS

ignisp is a minimal Lisp derivative designed for maximum portability and longevity. It is not a product, a library, or a learning exercise. It is a life project -- a computational environment designed to survive decades and be rebuildable on any platform that exists or will exist.

The name comes from Ignacio (Nacho's name) + Latin *ignis* (fire). Fire is the spark that bootstraps everything -- the kernel is the spark, the language is the flame. It is both deeply personal (his name) and structurally meaningful (the bootstrap spark).

The core philosophy:

- **The spec is permanent. The implementation is disposable.** In 50 years, the kernel may have been rewritten 20 times in 20 host languages. The spec is what stays constant. The value is in the spec and the library, not in the implementation.

- **Minimal kernel, maximal self-implementation.** The host provides the smallest possible set of primitives (~15). Everything else -- cons cells, reader, eval, GC, macros, object system -- is implemented in ignisp itself.

- **Absolute portability over performance.** 100x slowdown is acceptable. The kernel must be small enough to rewrite in a day in any host language that has arrays and basic arithmetic.

- **KISS and DRY.** Every added complexity must justify itself. Solo project principles that have paid dividends.

- **No hard dependencies.** If a dependency is small enough to rewrite in a day, it's acceptable. Otherwise, eliminate it.

- **The 50-year horizon.** This is not a weekend project. It is a language to grow with, modify at will, and rebuild on any future platform. When Nacho is 70, he wants to still be using this language.

* ARCHITECTURE: LAYERED DESIGN

The architecture is a layered bootstrap:

```
Layer 3: User programs, CLOS-like system, LOOP, FORMAT, etc.
         (All written in ignisp, portable)

Layer 2: Standard library
         (cons, car, cdr, reader, printer, eval, defmacro, macros,
          GC, string ops, list utilities -- all in ignisp, portable)

Layer 1.5: VM + Compiler (optional, for performance)
         (Register-based VM, instruction dispatch, register allocator
          -- all in ignisp, portable)

Layer 1: KERNEL -- implemented in host language
         (C today, Python tomorrow, Verilog in 10 years)
         ~200-500 lines. Provides ~15 primitives + 8 special forms.

Layer 0: HOST (C, Python, CL, Verilog, whatever the future brings)
```

Key architectural decisions:

1. **GC is a Layer 2 concern, not Layer 1.** In the array-based VM, GC is pure Lisp (~30-50 lines) because all state is in explicit arrays. The host provides allocation; GC is implemented in the standard library using only array operations. Mark-sweep is the chosen algorithm -- simpler than reference counting (no cycle problem) and simpler than Rust-style ownership (no compile-time analysis needed, preserves dynamic features).

2. **The compiler is Layer 2, not Layer 1.** The compiler is written in ignisp. It is automatically portable. Multiple backends possible (bytecode, C, x86). The interpreter is all you need to bootstrap.

3. **Object representation: tagged values in array slots.** Fixnums: low bit = 0, shift right. Heap pointers (indices): low bit = 1, clear bit. eq becomes = on raw values. This is how real Lisps do tagged representation, just with array indices instead of machine pointers.

4. **Two-phase bootstrap:** Phase 1: tree-walking interpreter (get language working, iterate on design). Phase 2: register-based VM (performance, structure, hardware-readiness). Layer 2 code runs unchanged on either execution engine.

5. **Variable register count per target.** The register count is a target parameter (8 on microcontroller, 16-32 on laptop, 256 on server). Same instruction set, same VM code. Only the compiler's register allocator changes behavior. Source portable, bytecode target-specific.

* MEMORY MANAGEMENT DECISIONS

- **Rust-style ownership was rejected** because it requires static analysis, restricts dynamic features (eval, runtime macros, arbitrary aliasing), and the result wouldn't be Lisp anymore. The complexity moves from runtime to compiler, and for a one-person project, compiler complexity is worse.

- **Reference counting was rejected** because cycles leak (need separate cycle collector), every assignment needs inc/dec, and it's more code than mark-sweep for less benefit.

- **Mark-sweep GC was chosen** because it's ~30-50 lines of Lisp, handles closures, cycles, eval, and dynamic dispatch with no language restrictions. The root set is explicit: global environment + VM call stack + temporary root stack. All are arrays the VM owns.

- **Phase 1 (current): bump allocator, no GC.** malloc everything, never free. Acceptable for short programs and REPL sessions. Add GC when memory pressure becomes real.

* THE KERNEL (Layer 1)

The kernel is the only host-dependent code. It must be small enough to rewrite in a day.

Primitives (host-provided, ~15):
- make-array, aref, aset (array operations)
- +, -, *, / (arithmetic)
- <, >, = (comparison)
- read-char, write-char (character I/O)
- error (abort)
- eq (identity/equality -- may be = on tagged values)

Special forms (8):
- if, quote, lambda, let, setq, defmacro, define, begin

The kernel does NOT provide: cons, car, cdr, strings, the reader, the printer, numbers beyond fixnum, garbage collection, cond, and, or, when, let*, labels, type predicates. All of these are built in Layer 2.

* THE VM (Layer 1.5, Future)

Register-based architecture (like Lua 5.x, Dalvik). ~20-25 instructions. The instruction set is called IGASM (Ignisp Assembly). Instructions reference registers by index. The VM dispatch loop is a Lisp function with TCO -- a state machine with state in arrays.

The entire VM state lives in host-provided arrays: the heap, registers, stack, and free list. No hidden state in C local variables. This means: GC rooting is trivial, serialization is free (image-based persistence like SBCL/Smalltalk), debugging is transparent, and portability is absolute.

* SPECIFICATION PHILOSOPHY

Nacho wants specs written in a mathematical/structural style, not tied to concrete encodings. Character assignments are implementation parameters with a reference assignment. The structure is permanent; the encoding is disposable.

Example: Instead of writing "space (U+0020) is whitespace", write "the whitespace class separates tokens" and separately document "in the reference assignment, whitespace = space, tab, newline, CR".

The spec has three parts:
1. **Kernel Spec** -- what the host provides. Frozen. The portable contract.
2. **Language Spec** -- reader, evaluator, types, scoping. Frozen once stable.
3. **Standard Library Spec** -- everything built on top. Grows forever.

* CURRENT STATE (v0.1 ALPHA)

A pilot implementation exists at `/var/home/nacho/repos/ignisp/ignisp.c` (~500 lines of C). It is a throwaway to see the language working end-to-end. It includes:

- Reader with strings, numbers, symbols, comments, quote macro, dotted pairs
- Printer with proper list and dotted pair rendering
- Evaluator with 8 special forms and 21 primitives
- Macros with rest parameters
- Lexical closures with captured environment
- Recursion support (define pre-binds name)
- Arrays with aref/aset
- Error recovery via setjmp/longjmp
- A working REPL

Known v0.1 limitations (documented, not bugs):
- No GC (leaks memory)
- No TCO (stack overflow on deep recursion)
- Closures don't support rest parameters (only macros do)
- No type checking on arithmetic
- No type predicates
- Strings don't round-trip (printer doesn't escape)
- No character type (fixnum ASCII codes)
- No multiple values
- No error recovery mechanism (just REPL reset)
- setq on unbound creates globals
- define always binds globally
- No modules/packages
- Fixed buffer sizes (255 char symbols, 1023 char strings)
- No file I/O

Files in `/var/home/nacho/repos/ignisp/`:
